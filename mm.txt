# MM & Adapter GTP Logic Reference

## MM Game Loop (game.go playOneGame)

### genmove response handling

```
resp = genmove(side)
if resp starts with "?":
    if contains "timeout" → ErrTimeout
    elif "RESIGN" → ErrResign (from engine)
    else → ErrResign (e.g. "? wrong color")
if move == "PASS":
    if legal_moves != 0 → ErrIllegalPass
    // No play command sent — pass doesn't change board.
    pass_counter++; if 2 → game over
    toggle side, continue
// Normal move:
sq = parse(move)
if sq illegal on framework board → ErrIllegalMove
board.ApplyMove(sq)
if board.IsOver() → break (natural end)
// Tell opponent about the move:
resp = send(opponent, "play <side> <move>")
if resp starts with "?" → ErrInvalidResponse (opponent malfunction)
toggle side
```

### Key MM rules

1. Framework board is AUTHORITATIVE. No "trust the engine."
2. Play response IS checked (was fire-and-forget, fixed 2026-06-30).
3. No `play <side> pass` sent — pass doesn't change board, opponent
   infers from next genmove.
4. Error codes: 0=None 1=illegal_move 2=timeout 3=crash 4=resign
   5=invalid_response 6=illegal_pass 7=opening_rejected

## Adapter Reference

Each adapter handles four GTP commands: play, genmove (normal), genmove (pass),
and the internal state transition after each.

### Invariants every adapter MUST follow

1. **play <color> <move>**: Apply the move for `<color>`. Do NOT validate
   color against internal side_to_move — the engine receives its own moves
   via genmove, not play. Play is always the opponent's move.

2. **play: after applying**: Just flip to the next player. Do NOT auto-pass
   for the opponent. The MM will send genmove to the opponent, who will
   pass if needed.

3. **genmove pass**: When engine has no legal moves, return PASS AND flip
   side_to_move (call make_pass / play_move(PASS)). Without the flip,
   the engine stays on a player who can't move, causing divergence.

4. **genmove normal move**: Play the move, flip to opponent. Then check:
   can the opponent move? If not, auto-pass for them (flip back to self).
   This keeps side_to_move in sync — the MM will send genmove to the
   opponent, who will pass, then back to us. We're already on our turn.

Key fix: Auto-pass for opponent moved from `game_state.make_move()` (shared
by both play and genmove) to `handle_genmove()` only. This prevents the
play handler from auto-passing, which was putting the engine one move ahead
of the MM and causing `? wrong color` on the next genmove.

### Darwersi (Arena/darwersi-gtp.c)

| Operation | Implementation | OK? |
|-----------|---------------|-----|
| play color check | None — just checks `DWIsPlayable(&gs, player, gp)` | ✅ |
| play apply | `DWPlay(&gs, player, gp)` — applies for specified player | ✅ |
| play after apply | Player flipped by DWPlay | ✅ |
| genmove pass | `mobilityNb[player] == 0` → `printf("= PASS")` → continue | ❌ does NOT flip player! |
| genmove move | `DWPlay(&gs, player, gp)` — applies move | OK |
| genmove opponent check | None — no auto-pass for opponent | ⚠️ missing |

Darwersi issues:
1. **Pass path doesn't flip player.** After returning PASS, internal state
   stays on the passing player. However, darwersi doesn't validate color
   on genmove or play, and `mobilityNb[player]` checks the specified
   player's mobility regardless of current turn — so this doesn't cause
   errors in practice. Still, should be fixed for correctness.
2. **No opponent auto-pass after own move.** If darwersi plays a move that
   leaves the opponent with no legal moves, side_to_move stays on opponent.
   The opponent will pass via genmove, then the MM will send genmove back
   to darwersi. Since darwersi doesn't validate color, it works. But the
   engine's internal turn is one step behind the game flow.
3. **Dead code** at line 265-266: `printf("= nodes 0...")` after `continue`
   is unreachable.
